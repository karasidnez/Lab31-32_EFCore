## 2. Полезные команды `dotnet ef`

| Команда | Описание |
|---------|----------|
| `dotnet ef migrations add ИмяМиграции` | Создаёт новую миграцию на основе изменений в моделях |
| `dotnet ef database update` | Применяет все неприменённые миграции к БД |
| `dotnet ef migrations list` | Показывает список миграций и их статус (применена/не применена) |
| `dotnet ef migrations remove` | Удаляет последнюю миграцию (только если она ещё не применена) |
| `dotnet ef database update ПредыдущаяМиграция` | Откатывает БД до указанной миграции |
| `dotnet ef migrations script` | Генерирует SQL-скрипт миграций без применения к БД |
| `dotnet ef --version` | Выводит версию установленного инструмента `dotnet-ef` |

## 6. Сравнительная таблица: LINQ vs SQL

| Операция | LINQ (C# + EF Core) | SQL (генерируется EF Core) |
|----------|----------------------|-----------------------------|
| Получить все задачи | `context.Tasks.ToListAsync()` | `SELECT * FROM Tasks` |
| Фильтрация по полю | `Where(t => t.IsCompleted == false)` | `WHERE IsCompleted = 0` |
| Поиск по подстроке | `Where(t => t.Title.Contains("SQL"))` | `WHERE Title LIKE '%SQL%'` |
| Сортировка | `OrderByDescending(t => t.CreatedAt)` | `ORDER BY CreatedAt DESC` |
| Пагинация | `Skip(5).Take(10).ToListAsync()` | `LIMIT 10 OFFSET 5` |
| Группировка + подсчёт | `GroupBy(t => t.Priority).Select(g => new { g.Key, Count = g.Count() })` | `SELECT Priority, COUNT(*) FROM Tasks GROUP BY Priority` |
| Агрегация (COUNT) | `CountAsync(t => t.IsCompleted)` | `SELECT COUNT(*) FROM Tasks WHERE IsCompleted = 1` |
| Вставка | `context.Tasks.Add(task); await context.SaveChangesAsync();` | `INSERT INTO Tasks (...) VALUES (...);` |
| Обновление | `task.Title = "new"; await context.SaveChangesAsync();` | `UPDATE Tasks SET Title = 'new' WHERE Id = ...` |
| Удаление | `context.Tasks.Remove(task); await context.SaveChangesAsync();` | `DELETE FROM Tasks WHERE Id = ...` |

## 7. Итоговая сравнительная таблица: хранение в памяти vs EF Core + SQLite

| Концепция | Хранение в памяти (`static List<T>`) | EF Core + SQLite |
|-----------|--------------------------------------|------------------|
| **Хранение данных** | `static List<T>` в RAM | Файл `.db` на диске |
| **После перезапуска** | Данные пропадают | Данные сохраняются |
| **Поиск по условию** | LINQ to Objects | LINQ to Entities → SQL |
| **Создание структуры** | Не нужно | Миграции (`dotnet ef`) |
| **Начальные данные** | Хардкод в коде | `HasData()` в миграции |
| **Получение данных** | `list.FirstOrDefault(...)` | `await db.Table.FindAsync(id)` |
| **Добавление** | `list.Add(item)` | `db.Table.Add(item)` + `SaveChangesAsync()` |
| **Удаление** | `list.Remove(item)` | `db.Table.Remove(item)` + `SaveChangesAsync()` |
| **Масштабируемость** | Ограничена RAM | Гигабайты данных |
| **Транзакции** | Нет | Встроены в EF Core |

## 8. Главные выводы 

1. **EF Core – это "переводчик" между C# и SQL.**  
   Вместо написания SQL-строк вручную мы пишем LINQ-запросы, а EF Core сам генерирует оптимальный SQL. Это снижает количество ошибок и ускоряет разработку.

2. **Миграции – это система контроля версий для структуры БД.**  
   Как Git для кода, миграции позволяют отслеживать, кто, когда и какие изменения вносил в схему базы данных. Легко откатиться назад или применить обновления на другом компьютере.

3. **`SaveChangesAsync()` – ключевой момент.**  
   До вызова этого метода все изменения (добавление, обновление, удаление) живут только в памяти и не влияют на файл БД. Только после `SaveChangesAsync()` транзакция фиксируется и данные становятся постоянными.

4. **Использование `async`/`await` при работе с БД – стандарт.**  
   Запросы к диску или сети – долгие операции. Блокировать поток сервера в ожидании БД – плохая практика. `async`/`await` позволяет серверу обрабатывать другие запросы, пока один ожидает ответа от БД, повышая масштабируемость.

5. **LINQ позволяет гибко строить запросы.**  
   Благодаря отложенному выполнению (`AsQueryable()`, `Where()`, `Select()`) мы можем добавлять условия фильтрации по мере поступления параметров от клиента, а реальный SQL-запрос уйдёт в БД только при вызове `ToListAsync()`, `CountAsync()` и т.д..
